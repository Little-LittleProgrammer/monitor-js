# Quantum Monitor 新人上手指南

## 📖 项目简介

Quantum Monitor 是一款功能完善的前端监控 SDK，支持浏览器端和微信小程序端。通过插件化架构实现了错误监控、性能监控、行为监控等核心功能。

### 核心特性

- ✅ **错误监控**：JS 错误、资源错误、Promise 错误、API 错误
- ✅ **性能监控**：FP、FCP、LCP、FID、CLS、导航时序、资源加载
- ✅ **行为监控**：用户点击、路由变化、PV 统计
- ✅ **多端支持**：浏览器、微信小程序
- ✅ **插件化架构**：功能模块化，按需加载
- ✅ **配置化设计**：所有功能可独立开关

## 🔑 核心原理

### 1. 发布-订阅模式

整个 SDK 基于发布-订阅模式构建，实现插件间的解耦通信。

**核心类：Subscribe**

```typescript
class Subscribe<T> {
  private cache: Map<T, MonitorCallback[]>;
  
  // 订阅事件
  on(eventName: T, callback: MonitorCallback): void {
    const callbacks = this.cache.get(eventName) || [];
    callbacks.push(callback);
    this.cache.set(eventName, callbacks);
  }
  
  // 发布事件
  emit<D>(eventName: T, data: D): void {
    const callbacks = this.cache.get(eventName);
    callbacks?.forEach(callback => callback(data));
  }
}
```

**工作流程：**

```
插件监听事件 → emit 发布数据 → transform 转换 → consumer 消费
```

### 2. 插件系统

**插件接口定义：**

```typescript
interface BasePluginType {
  name: string;              // 插件名称
  type: string;              // 插件类型（error/performance/behavior）
  monitor: (emit) => void;   // 监听器：收集原始数据
  transform?: (data) => any; // 转换器：格式化数据
  consumer?: (data) => void; // 消费者：处理数据（上报/缓存）
}
```

**三阶段处理：**

| 阶段 | 职责 | 示例 |
|------|------|------|
| **monitor** | 绑定事件监听，收集原始数据 | 监听 window.onerror |
| **transform** | 格式化为标准上报格式 | 提取错误堆栈、生成 errorUid |
| **consumer** | 处理数据（上报/添加面包屑） | 调用 report.send() |

**插件加载流程：**

```typescript
class BaseClient {
  use(plugins: BasePluginType[]): void {
    const subscribe = new Subscribe();
    
    plugins.forEach(plugin => {
      // 1. 检查插件是否启用
      if (!this.isPluginEnable(plugin.name)) return;
      
      // 2. 调用 monitor，传入 emit 函数
      plugin.monitor.call(this, subscribe.emit.bind(subscribe));
      
      // 3. 构建处理链：transform → consumer
      const handler = (data) => {
        const transformed = plugin.transform?.call(this, data);
        plugin.consumer?.call(this, transformed);
      };
      
      // 4. 订阅事件
      subscribe.on(plugin.name, handler);
    });
  }
}
```

### 3. 数据上报机制

**四层数据处理：**

```
原始数据 → 格式化 → 添加基础信息 → 添加平台信息 → 上报
```

**上报策略：**

1. **批量上报**：数据缓存达到阈值（默认 50 条）时批量发送
2. **实时上报**：错误类数据立即上报（isImmediate=true）
3. **页面卸载上报**：beforeunload 事件触发时上报剩余数据

**三种上报方式（级联）：**

| 方式 | 优先级 | 特点 | 使用场景 |
|------|--------|------|----------|
| **Beacon API** | 1 | 不阻塞页面，可靠性高 | 页面卸载时 |
| **XMLHttpRequest** | 2 | 支持自定义请求头 | 正常上报 |
| **Image** | 3 | 兼容性最好 | 兜底方案 |

### 4. 用户行为栈（Breadcrumb）

**作用：** 记录用户操作轨迹，错误发生时附加到错误数据中，帮助复现问题。

**数据结构：**

```typescript
interface BreadcrumbData {
  type: string;        // 行为类型（CLICK/ROUTE/XHR/ERROR）
  data: any;          // 行为数据
  level: string;      // 严重程度（Info/Warning/Error）
  time: number;       // 时间戳
}
```

**特性：**
- 最多保留 10 条记录（可配置）
- 自动按时间排序
- 支持 beforePushBreadcrumb 钩子过滤

```typescript
class BrowserClient extends BaseClient {
  report: BrowserReport;
  
  // 插件启用检查
  isPluginEnable(name: string): boolean {
    const flag = `disabled${formatString(name)}`;  // 如：disabledJsError
    return !this.options[flag];

    const flag = `disabled${firstStrToUppercase(type)}`;  // 如：disabledError
    return !this.options[flag];
```

#### BrowserReport

```typescript
class BrowserReport extends BaseReport {
  // 三种上报方式
  beaconPost(data, url): void;  // Beacon API
  post(data, url): void;        // XMLHttpRequest
  imgRequest(data, url): void;  // Image
  
  // 智能选择上报方式
  report(data, url): void {
    const fn = this.useImgUpload ? this.imgRequest :
               isSupportSendBeacon() ? this.beaconPost : this.post;
    
    // 使用 requestIdleCallback 避免阻塞主线程
    requestIdleCallback(() => fn(data, url));
  }
}
```

## 🔍 关键实现原理

### 1. 错误监控原理

#### JS 错误捕获

```typescript
const jsErrorPlugin = {
  name: 'JsError',
  monitor(emit) {
    window.addEventListener('error', (e: ErrorEvent) => {
      if (!e.target.localName) {  // 排除资源错误
        emit('JsError', e);
      }
    }, true);  // 捕获阶段
  },
  transform(errorEvent) {
    return {
      type: 'error',
      subType: 'JsError',
      mainData: {
        errorUid: getErrorUid(`JE-${errorEvent.message}-${errorEvent.filename}`),
        msg: errorEvent.message,
        stack: errorEvent.error?.stack,
        meta: {
          file: errorEvent.filename,
          line: errorEvent.lineno,
          column: errorEvent.colno
        }
      }
    };
  },
  consumer(data) {
    this.report.breadcrumb.push({...});  // 添加到行为栈
    this.report.send(data, true);        // 立即上报
  }
};
```

**关键点：**
- 使用 `errorUid` 去重（基于错误类型、消息、文件名）
- 捕获阶段监听（`useCapture: true`）确保优先捕获
- 立即上报（`isImmediate: true`）

#### XHR/Fetch 拦截

```typescript
// XHR 拦截原理
const _open = XMLHttpRequest.prototype.open;
const _send = XMLHttpRequest.prototype.send;

XMLHttpRequest.prototype.open = function(...args) {
  this.url = args[1];
  this.method = args[0];
  this.startTime = Date.now();
  _open.apply(this, args);
};

XMLHttpRequest.prototype.send = function(...args) {
  this.addEventListener('loadend', () => {
    const duration = Date.now() - this.startTime;
    emit('XHR', {
      url: this.url,
      method: this.method,
      status: this.status,
      duration
    });
  });
  _send.apply(this, args);
};
```

### 2. 性能监控原理

#### PerformanceObserver API

```typescript
const lcpPlugin = {
  monitor(emit) {
    const observer = new PerformanceObserver((list) => {
      observer.disconnect();  // LCP 只需要最后一次
      for (const entry of list.getEntries()) {
        emit('LCP', entry.toJSON());
      }
    });
    observer.observe({ 
      type: 'largest-contentful-paint', 
      buffered: true  // 获取历史数据
    });
  }
};
```

#### CLS 计算（会话窗口）

```typescript
let sessionValue = 0;
let sessionEntries = [];

observer.observe({ type: 'layout-shift', buffered: true });

// 会话窗口判断：
// 1. 与前一个偏移间隔 < 1 秒
// 2. 与第一个偏移间隔 < 5 秒
if (entry.startTime - lastEntry.startTime < 1000 &&
    entry.startTime - firstEntry.startTime < 5000) {
  sessionValue += entry.value;
  sessionEntries.push(entry);
}
```

#### 导航时序计算

```typescript
const navigationPlugin = {
  transform(entry) {
    return {
      tti: entry.domComplete - entry.fetchStart,              // 首次可交互
      domReady: entry.domContentLoadedEventEnd - entry.fetchStart,
      load: entry.loadEventEnd - entry.fetchStart,            // 总加载时间
      dns: entry.domainLookupEnd - entry.domainLookupStart,   // DNS 查询
      tcp: entry.connectEnd - entry.connectStart,             // TCP 连接
      ssl: entry.connectEnd - entry.secureConnectionStart,    // SSL 握手
      ttfb: entry.responseStart - entry.requestStart,         // 首字节时间
      trans: entry.responseEnd - entry.responseStart,         // 传输时间
      domParse: entry.domInteractive - entry.responseEnd,     // DOM 解析
      res: entry.loadEventStart - entry.domContentLoadedEventEnd // 资源加载
    };
  }
};
```

### 3. 路由监控原理

#### History 路由劫持

```typescript
const historyRoutePlugin = {
  monitor(emit) {
    const _pushState = history.pushState;
    const _replaceState = history.replaceState;
    
    // 劫持 pushState
    history.pushState = function(...args) {
      _pushState.apply(this, args);
      emit('HistoryRoute', {
        from: oldURL,
        to: location.href
      });
    };
    
    // 劫持 replaceState
    history.replaceState = function(...args) {
      _replaceState.apply(this, args);
      emit('HistoryRoute', {
        from: oldURL,
        to: location.href
      });
    };
    
    // 监听 popstate（浏览器前进/后退）
    window.addEventListener('popstate', () => {
      emit('HistoryRoute', {
        from: oldURL,
        to: location.href
      });
    });
  }
};
```

## 📝 常见问题

### 1. 为什么使用 Monorepo？

**优势：**
- 代码复用：核心功能抽象为基类，各端继承
- 统一管理：依赖、构建、测试流程统一
- 独立发布：每个包可独立发版
- 类型共享：TypeScript 类型定义统一维护

### 2. 插件如何实现解耦？

通过发布-订阅模式：
- 插件只负责 emit 数据，不关心数据如何处理
- Subscribe 负责事件分发
- 各插件独立运行，互不影响

### 3. 如何避免重复上报？

**错误去重：**
```typescript
// 生成唯一 errorUid
const errorUid = getErrorUid(`${type}-${message}-${filename}`);

// 检查是否已上报
if (this.submitErrorUids.includes(errorUid)) {
  return;  // 跳过
}

// 记录已上报
this.submitErrorUids.push(errorUid);
```

### 4. 性能监控会影响页面性能吗？

**优化措施：**
- 使用 `requestIdleCallback` 延迟处理
- 批量上报减少网络请求
- PerformanceObserver 异步监听
- 插件按需加载

### 5. 如何调试插件？

